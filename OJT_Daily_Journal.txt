Week 3 (July 27, 2026)
---------------------------------
July 27 (Monday) — The main task today was fixing a critical login issue that
prevented users from signing into the POS system. After logging in, the app
would show "Login successful. Redirecting..." but never actually redirect,
requiring the user to click Sign In a second time. On top of that, Firestore
was returning "Missing or insufficient permissions" errors on every profile
read and write, even though the user was correctly authenticated.

After extensive debugging, the root cause was identified: the code was calling
Firestore (to read/write user profiles) before the Firebase Auth state had
fully propagated to the Firestore SDK. When signInWithEmailAndPassword
returns, the user object is available but the internal auth token that
Firestore uses has not been set yet. Firestore then sends requests without an
auth token, and the security rules reject them. The fix was to move all
routing logic into the watchAuth callback, which fires after onAuthStateChanged
ensures the auth state is fully synced with Firestore. Additionally, a session
date race condition was found — onAuthStateChanged fired before the session
date was set in localStorage, causing the app to immediately log the user
back out. This was fixed by setting the session date before calling
loginWithEmail.

Other fixes applied during this session:
- Removed ensureDefaultAccounts(), which tried to write to Firestore before
  any user was signed in, causing permission denials that got cached.
- Disabled persistent Firestore cache to prevent stale cached denials from
  poisoning future requests.
- Updated Firestore security rules to allow admins to read, create, and update
  all user documents in the users collection, fixing the accounts directory
  in the admin portal.
- Removed unused imports (createAuthUserByAdmin, fetchSignInMethodsForEmail).
- Added a routingInProgress guard to prevent double navigation from concurrent
  watchAuth and window.login calls.

Later in the day, I improved the Employee Order and PWD/Senior discount
checkboxes on the POS interface. The original implementation had a
double-toggle bug: the <label> element had an onclick handler that toggled
the state, but the native browser label-to-checkbox behavior also toggled
it, causing the two to cancel each other out. This meant clicking anywhere
on the label text did nothing — users had to click the tiny checkbox itself.

Fixes applied:
- Moved onclick from <label> to onchange on <input type="checkbox"> for both
  the Employee Order and PWD/Senior toggles, letting the native label
  behavior handle the toggle and only reading the state on change.
- Added mutual exclusivity: when one discount is checked, the other is
  automatically unchecked, disabled, and visually dimmed (opacity 0.45 with
  pointer-events: none). Unchecking re-enables the other.
- Added user-select: none to the discount toggle labels to prevent accidental
  text selection when clicking.
- Fixed clearCart and moveCurrentOrderToUnpaid to properly reset both
  discount states and re-enable both checkboxes (previously only PWD/Senior
  was reset, leaving Employee Order stuck in a checked/disabled state).
- Fixed the payment success reset to also re-enable both checkboxes.
- Fixed the updateCart empty-cart early return to hide discount rows
  (previously the discount row remained visible after clearing the cart
  because updateCart returned before reaching the hide/show logic).
- Fixed buildUnpaidOrderFromCart to save isEmployeeOrder (previously only
  isPwdSenior was saved, so restoring an unpaid employee order lost that
  state).
- Fixed restoreUnpaidOrderToCart to restore both discount states and apply
  mutual exclusivity when restoring from unpaid orders.

                    SECURITY AUDIT FINDINGS (July 22, 2026)
================================================================================

FIXED:
- Removed hardcoded admin credentials (email/password) from source code
- Removed auto-provisioning of default admin account (ensureDefaultAdminAccount)
- Fixed session expiry: auth session now resets at midnight (date-based check)
- Fixed privilege escalation: users without a role document are now denied
  access instead of being silently promoted to admin
- Fixed Firestore rules: removed isLegacyAdmin() which granted admin access
  to any signed-in user without a role document
- Fixed auth race condition: Firestore calls fired before auth state propagated
- Fixed session date race condition: date set before login to prevent logout
- Fixed accounts directory: admin can now read all user documents

OPEN -- XSS VULNERABILITIES (16 sites across 4 files):
Firestore-sourced data (menu names, category names, variant names, addon
names, order IDs) is interpolated into innerHTML templates without using
escapeHtml(). If an admin account were compromised, malicious data could
execute scripts in the browser.
